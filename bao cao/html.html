STAYZ PROJECT STUDY GUIDE#

Ghi chú đối chiếu source: File được tạo từ nội dung phân tích do người dùng cung cấp. Trong phiên làm việc này không có toàn bộ thư mục source StayZ để mở và xác minh trực tiếp, vì vậy các đường dẫn file được giữ nguyên theo tài liệu. Không có secret thật nào được đưa vào trang.

1. StayZ là gì?#

StayZ là ứng dụng đặt phòng khách sạn đa nền tảng được xây dựng bằng Flutter. Ứng dụng cho phép người dùng tìm kiếm nơi lưu trú, xem khách sạn và phòng, kiểm tra tình trạng phòng theo ngày, đặt phòng, thanh toán qua PayOS, quản lý booking, yêu thích khách sạn, nhận thông báo, đánh giá sau chuyến đi và nhận đề xuất bằng AI.

Hệ thống gồm hai phần:

  • Frontend mobile/web: Flutter và Dart.
  • Backend REST API: Node.js, Express và MongoDB.
  • Các dịch vụ hỗ trợ: Redis, Socket.IO, Cloudinary, PayOS, Gmail OAuth, Google OAuth và OpenAI API.

Bằng chứng:

  • pubspec.yaml
  • lib/main.dart
  • lib/app/app.dart
  • backend/stayz_api/package.json
  • backend/stayz_api/server.js

Câu trả lời mẫu khi thầy hỏi “StayZ là gì?”Trả lời mẫu#

“StayZ là ứng dụng hỗ trợ người dùng tìm kiếm và đặt phòng khách sạn. Frontend được phát triển bằng Flutter để có thể chạy trên nhiều nền tảng. Backend sử dụng Node.js và Express, dữ liệu được lưu bằng MongoDB. Ngoài các chức năng đặt phòng cơ bản, hệ thống còn tích hợp PayOS để thanh toán, Redis để hỗ trợ rate limit và chống đặt trùng phòng, Cloudinary để lưu hình ảnh, Socket.IO cho chat realtime và OpenAI để tư vấn khách sạn dựa trên dữ liệu thật của hệ thống.”

2. Kiến trúc tổng thể#

Kiến trúc thực tế của StayZ có thể mô tả như sau:

Người dùng
→ Flutter Page/Widget
→ Repository hoặc AuthService
→ ApiService
→ HTTP REST API
→ Express Router
→ Middleware
→ Controller
→ Business Service
→ Mongoose Model
→ MongoDB

Các dịch vụ bên ngoài được backend gọi khi cần:

  • Redis: rate limiting, blacklist token và distributed lock.
  • PayOS: tạo liên kết thanh toán và xử lý webhook.
  • Cloudinary: upload ảnh avatar, khách sạn và phòng.
  • Gmail OAuth: gửi OTP.
  • Google OAuth: đăng nhập bằng tài khoản Google.
  • OpenAI: sinh câu trả lời tư vấn từ dữ liệu StayZ.
  • Socket.IO: nhắn tin thời gian thực.

Vai trò từng tầng#

Flutter Page/Widget#

Hiển thị giao diện, nhận thao tác và điều hướng. Page không nên trực tiếp xử lý MongoDB hoặc nghiệp vụ backend.

Ví dụ:

  • lib/features/search/presentation/pages/search_page.dart
  • lib/features/booking/presentation/pages/payment_checkout_page.dart
  • lib/features/profile/presentation/pages/settings_page.dart

Repository#

Là lớp trung gian giữa UI và API. Repository gọi endpoint rồi chuyển JSON thành model Dart.

Bằng chứng:

  • lib/shared/repositories/stayz_repository.dart
  • lib/features/admin/data/admin_repository.dart

ApiService#

Chuẩn hóa GET, POST, PUT, PATCH, DELETE, multipart, bearer token, timeout và lỗi mạng.

Bằng chứng:

  • lib/services/api_service.dart

ApiService dùng timeout 20 giây. Nó chuyển lỗi HTTP thành ApiException, hỗ trợ tiếng Việt/Anh và xử lý upload theo MIME type.

Router#

Định nghĩa URL, HTTP method và middleware.

Ví dụ:

  • backend/stayz_api/src/routes/booking.router.js
  • backend/stayz_api/src/routes/payment.router.js

Controller#

Nhận req, lấy params/body/user, gọi service và trả response.

Service backend#

Chứa nghiệp vụ thật như kiểm tra phòng trống, tính tiền, kiểm tra quyền, tạo payment và cập nhật trạng thái.

Ví dụ:

  • backend/stayz_api/src/services/booking.service.js
  • backend/stayz_api/src/services/payment.service.js
  • backend/stayz_api/src/services/ai.service.js

Mongoose Model#

Khai báo cấu trúc MongoDB, enum, quan hệ, index và giá trị mặc định.

Bằng chứng:

  • backend/stayz_api/src/models/

Vì sao cần chia tầng?#

Nếu toàn bộ xử lý nằm trong Flutter Page hoặc Express Controller:

  • Code khó đọc.
  • Khó tái sử dụng.
  • Khó kiểm thử.
  • Dễ sai nghiệp vụ giữa nhiều màn hình.
  • Khó thay API hoặc database.
  • UI phải biết quá nhiều chi tiết backend.

StayZ đã tách UI, repository, API service, route, controller, service và model, dù một số file vẫn còn khá lớn.

3. Entry point và khởi động ứng dụng#

Entry point Flutter là:

  • lib/main.dart

Quá trình khởi động:

  1. Gọi WidgetsFlutterBinding.ensureInitialized().
  2. Tải ngôn ngữ đã lưu bằng AppLocale.instance.load().
  3. Chạy StayZApp.
  4. MaterialApp bắt đầu tại AppRoutes.authGate.

Bằng chứng:

  • lib/main.dart
  • lib/app/app.dart
  • lib/app/routes/app_routes.dart

AuthGatePage quyết định:

  • Người dùng chưa xem onboarding → onboarding.
  • Chưa đăng nhập → login.
  • Có access token còn hạn → home.

Ứng dụng kiểm tra trường exp trong JWT ở phía Flutter. Nếu token đã hết hạn, dữ liệu đăng nhập được xóa.

Bằng chứng:

  • lib/features/auth/presentation/pages/auth_gate_page.dart
  • lib/services/auth_service.dart

Project có cấu hình Android, iOS, web, Windows và macOS. Trọng tâm sản phẩm là ứng dụng đặt phòng trên mobile.

Bằng chứng:

Bằng chứng:

SharedPreferences#

Dùng lưu dữ liệu nhỏ trên thiết bị:

  • Đã xem onboarding hay chưa.

  • User ID.
  • Email.

Bằng chứng:

  • lib/features/admin/presentation/pages/admin_dashboard_page.dart

Lưu ý bảo mật: SharedPreferences tiện nhưng không phải secure storage chuyên dụng. Với production, token nên được cân nhắc lưu bằng secure storage.

qr_flutter#

5. Công nghệ backend#

Node.js và Express#

Express xử lý REST API, middleware, CORS, JSON body và static files.

Bằng chứng:

  • backend/stayz_api/server.js
  • backend/stayz_api/package.json

MongoDB và Mongoose#

MongoDB lưu user, property, room, booking, payment, review, favorite, notification và chat.

Mongoose khai báo schema, validation, enum, reference và index.

bcrypt#

Hash mật khẩu trước khi lưu. Backend không nên lưu mật khẩu dạng plain text.

JSON Web Token#

Backend tạo access token và refresh token.

Access token được gửi bằng header:

Authorization: Bearer <token>

Middleware protect xác minh JWT, kiểm tra blacklist và tải lại role hiện tại từ database.

Bằng chứng:

  • backend/stayz_api/src/utils/token.util.js
  • backend/stayz_api/src/middlewares/protect.middleware.js

Redis và ioredis#

Redis có ba vai trò chính:

  1. Rate limiting.
  2. Blacklist access token khi logout.
  3. Hỗ trợ distributed lock trong booking.

Redis được thiết kế theo hướng fail-open cho một số thao tác: nếu Redis lỗi, request không nhất thiết bị chặn hoàn toàn.

Bằng chứng:

  • backend/stayz_api/src/config/redis.config.js
  • backend/stayz_api/src/middlewares/rateLimit.middleware.js
  • backend/stayz_api/src/middlewares/protect.middleware.js

Redlock#

Dùng tạo lock phân tán theo room và khoảng ngày để giảm nguy cơ hai request đồng thời cùng đặt vượt số lượng phòng.

Bằng chứng:

  • backend/stayz_api/src/services/booking.service.js

Socket.IO#

Backend có:

  • Xác thực socket bằng JWT.
  • Join conversation.
  • Leave conversation.
  • Send message.
  • Broadcast message_received.
  • Typing và stop typing.

Bằng chứng:

  • backend/stayz_api/src/config/socket.config.js

Lưu ý quan trọng: pubspec.yaml hiện không khai báo package Socket.IO client. Vì vậy backend có hạ tầng realtime nhưng chưa nên khẳng định Flutter client đang sử dụng đầy đủ Socket.IO. AI chat hiện đi bằng HTTP.

Multer#

Nhận multipart upload.

Cloudinary#

Lưu ảnh cloud cho:

  • Avatar.
  • Ảnh chính property.
  • Gallery property.
  • Ảnh phòng.
  • Gallery phòng.

PayOS#

Tạo payment link, QR và webhook cập nhật trạng thái giao dịch.

Nếu thiếu PayOS key, SDK không được khởi tạo và backend cảnh báo payment flow có thể không hoạt động.

Bằng chứng:

  • backend/stayz_api/src/config/payos.config.js
  • backend/stayz_api/src/services/payment.service.js

Passport Google OAuth#

Backend hỗ trợ:

  • /auth/google
  • /auth/google/callback

Bằng chứng:

  • backend/stayz_api/src/routes/auth.router.js
  • backend/stayz_api/src/config/passport.config.js

Gmail OAuth#

Dùng gửi OTP đăng ký hoặc đặt lại mật khẩu.

OpenAI API#

Backend gọi OpenAI Chat Completions. Model mặc định là gpt-4o-mini, có thể đổi bằng OPENAI_MODEL.

Nếu không có API key, OpenAI lỗi hoặc timeout 12 giây, backend dùng câu trả lời fallback dựa trên dữ liệu StayZ.

Bằng chứng:

  • backend/stayz_api/src/services/ai.service.js

8. Luồng đăng ký#

Luồng source-confirmed:

  1. Người dùng nhập họ tên, email, số điện thoại và mật khẩu.
  2. Flutter gọi requestRegisterOtp.
  3. Backend tạo OTP và gửi email.
  4. Người dùng nhập OTP 6 chữ số.
  5. Flutter gọi verifyRegisterOtp.
  6. Khi OTP hợp lệ, Flutter gọi endpoint tạo user với register_code.
  7. Backend validate dữ liệu.
  8. Mật khẩu được hash bằng bcrypt.
  9. User được lưu vào MongoDB.

Endpoint:

  • POST /api/users/request-register-otp
  • POST /api/users/verify-register-otp
  • POST /api/users/create

Rate limit:

  • Request OTP: 3 lần/15 phút.
  • Verify OTP: 5 lần/15 phút.
  • Tạo tài khoản: 5 lần/15 phút.

Bằng chứng:

  • lib/features/auth/presentation/pages/register_page.dart
  • lib/features/auth/presentation/pages/otp_page.dart
  • lib/services/auth_service.dart
  • backend/stayz_api/src/routes/users.router.js

9. Luồng đăng nhập#

  1. Flutter gửi email và password.
  2. Backend tìm user.
  3. bcrypt so sánh mật khẩu.
  4. Backend trả access token, refresh token và user.
  5. Flutter hiện chỉ lưu access token cùng thông tin user.
  6. AuthGate kiểm tra token có tồn tại và còn hạn.
  7. Các API bảo vệ nhận access token qua Bearer header.

  • POST /api/users/login
  • POST /api/users/refresh-token
  • POST /api/users/logout

Điểm cần nói chính xác:

Backend có refresh-token API, nhưng Flutter AuthService hiện chủ yếu lưu access token và chưa thể hiện một cơ chế tự động refresh token hoàn chỉnh. Khi token hết hạn, phía Flutter logout và yêu cầu đăng nhập lại.

Quá trình PayOS#

  1. Booking pending được tạo.
  2. Flutter gọi: POST /api/payment/create/:bookingId
  3. Backend lấy booking thật.
  4. Backend tính lại payment quote.
  5. Backend tạo order code.
  6. PayOS trả checkout URL và QR.
  7. Payment được lưu với trạng thái pending.
  8. Người dùng thanh toán.
  9. PayOS gọi webhook.
  10. Backend xác minh webhook.
  11. Backend kiểm tra số tiền webhook phải khớp payment.amount.
  12. Payment chuyển thành PAID.
  13. Booking chuyển:
    • status = confirmed
    • payment_status = paid
  14. payment_expires_at được xóa.

Nếu giao dịch bị hủy, payment chuyển CANCELLED.

Điểm tốt:

  • Giá được tính ở server.
  • Webhook được xác minh.
  • Có kiểm tra amount.
  • Webhook đã xử lý payment paid theo hướng idempotent.
  • Chỉ payment thành công mới xác nhận booking.

Rủi ro cần nói:

  • Cần test thật với PayOS sandbox/production.
  • Trạng thái payment dùng cả lowercase pending và uppercase PAID, CANCELLED, chưa hoàn toàn đồng nhất.
  • Cần kiểm tra trường hợp webhook thành công nhưng cập nhật booking thất bại giữa chừng.
  • Cần reconciliation job nếu webhook bị bỏ lỡ.

25. Bảo mật#

Điểm tốt#

  • Password được hash bằng bcrypt.
  • JWT bảo vệ API.
  • Middleware tải role hiện tại từ database.
  • Có admin authorization.
  • Redis blacklist token logout.
  • Rate limit login, register và OTP.
  • Kiểm tra ownership booking/payment.
  • Giá booking được tính lại ở server.
  • Webhook kiểm tra số tiền.
  • Upload có kiểm soát MIME.
  • Favorite và review có unique index.
  • Distributed lock giảm overbooking.
  • Admin action có audit log.

Hạn chếHạn chế#

  • Flutter lưu token trong SharedPreferences thay vì secure storage.
  • Client chưa dùng refresh token đầy đủ.
  • CORS Socket.IO đang cho phép origin *.
  • Redis fail-open giúp hệ thống không chết nhưng làm giảm bảo vệ rate limit/lock khi Redis lỗi.
  • Cần test kỹ transaction giữa payment và booking.
  • Cần xác minh upload size/type ở runtime.
  • Không được khẳng định hệ thống an toàn tuyệt đối.
  • Cần rotation, quản lý và bảo vệ production secret.

Câu trả lời mẫuTrả lời mẫu#

“Dự án sử dụng bcrypt để hash mật khẩu, JWT để xác thực và middleware để phân quyền. Role admin được kiểm tra lại từ database. Redis hỗ trợ blacklist token, rate limit và lock booking. Giá thanh toán được tính ở server và webhook PayOS được xác minh. Tuy nhiên hệ thống vẫn còn các hướng cải thiện như chuyển token sang secure storage, hoàn thiện refresh token và tăng khả năng phục hồi transaction payment.”

30. CÂU HỎI VẤN ĐÁP VÀ TRẢ LỜICâu thầy dễ hỏi#

Câu 1: Vì sao chọn Flutter?#

Flutter cho phép dùng một codebase để phát triển nhiều nền tảng. Nó có hệ thống widget mạnh, hiệu năng UI tốt và phù hợp với ứng dụng mobile có nhiều màn hình như StayZ.

Câu 2: Vì sao chọn Node.js và Express?#

Node.js phù hợp với REST API và các tác vụ I/O như database, thanh toán, upload và chat. Express giúp tổ chức route, middleware, controller và service khá đơn giản.

Câu 3: Vì sao chọn MongoDB?#

Dữ liệu khách sạn và phòng có nhiều thuộc tính lồng nhau như amenities, gallery và trạng thái. MongoDB linh hoạt với dạng dữ liệu document và Mongoose hỗ trợ schema, validation, index và reference.

Câu 4: Repository khác Service thế nào?#

Repository phía Flutter chịu trách nhiệm lấy dữ liệu từ API và chuyển thành model. Service backend xử lý nghiệp vụ như tính giá, kiểm tra phòng trống và cập nhật trạng thái.

Câu 5: Controller khác Service thế nào?#

Controller làm việc với request/response. Service xử lý logic nghiệp vụ. Tách hai phần giúp service tái sử dụng và dễ test hơn.

Câu 6: Middleware là gì?#

Middleware chạy trước controller để xác thực JWT, kiểm tra admin, rate limit, upload hoặc audit.

Câu 7: JWT hoạt động thế nào?#

Sau login, backend ký JWT chứa định danh user. Flutter gửi token trong Bearer header. Backend xác minh chữ ký và lấy user hiện tại trước khi cho truy cập API bảo vệ.

Câu 8: Access token và refresh token khác nhau thế nào?#

Access token dùng gọi API và thường có thời hạn ngắn hơn. Refresh token dùng xin access token mới. Backend StayZ đã có refresh endpoint nhưng Flutter chưa sử dụng đầy đủ.

Câu 9: Redis dùng làm gì?#

Redis được dùng cho rate limit, blacklist token logout và hỗ trợ distributed lock khi đặt phòng.

Câu 10: Làm sao tránh đặt trùng phòng?#

Backend dùng Redlock để giới hạn xử lý đồng thời, sau đó đếm các booking overlap theo ngày và so sánh với room.quantity.

Câu 11: Nếu Redis lỗi thì sao?#

Một số thao tác Redis được thiết kế fail-open để API vẫn hoạt động. Tuy nhiên rate limit và lock có thể giảm hiệu lực, nên database validation vẫn rất quan trọng.

Câu 12: Giá booking được tính ở đâu?#

Giá được tính ở backend từ giá phòng trong database, số đêm và số phòng. Backend không tin giá client gửi lên.

Câu 13: Vì sao booking có trạng thái pending?#

Booking pending nghĩa là phòng đang được giữ trong thời gian người dùng chờ thanh toán. Code hiện giữ booking khoảng 15 phút.

Câu 14: Khi nào booking confirmed?#

Khi webhook PayOS hợp lệ xác nhận giao dịch thành công, payment chuyển PAID và booking chuyển confirmed.

Câu 15: Tại sao cần webhook?#

Người dùng có thể đóng app hoặc không quay lại return URL. Webhook cho phép PayOS thông báo trực tiếp cho backend về kết quả thanh toán.

Câu 16: Làm sao chống giả mạo số tiền?#

Backend tự tính payment amount và khi nhận webhook sẽ so sánh amount từ webhook với amount đã lưu.

Câu 17: Deposit 30% là gì?#

Người dùng thanh toán trước 30%, phần còn lại thanh toán tại khách sạn.

Câu 18: Full payment có gì khác?#

Người dùng thanh toán toàn bộ. Logic hiện tại áp dụng giảm 10% cho phương án này.

Câu 19: Socket.IO khác REST thế nào?#

REST hoạt động theo request-response. Socket.IO giữ kết nối hai chiều để server có thể đẩy message realtime.

Câu 20: Flutter đã dùng Socket.IO chưa?#

Backend có Socket.IO nhưng dependency Flutter hiện chưa có Socket.IO client. Vì vậy chưa nên khẳng định realtime chat đã hoàn chỉnh end-to-end.

Câu 21: AI có tự bịa khách sạn không?#

Thiết kế backend đưa dữ liệu property và room thật vào context. Nếu OpenAI lỗi, hệ thống dùng fallback từ database thay vì bịa dữ liệu.

Câu 22: Nếu không có OpenAI API key thì sao?#

Backend dùng fallbackReply để trả lời dựa trên dữ liệu StayZ.

Câu 23: Cloudinary dùng làm gì?#

Cloudinary lưu avatar, ảnh property và ảnh room trên cloud để frontend có URL truy cập ổn định.

Câu 24: Làm sao phân quyền admin?#

Flutter có AdminAccessGate. Backend dùng protect và adminOnly. Backend tải role hiện tại từ MongoDB nên không chỉ tin role cũ trong token.

Câu 25: Favorite bị thêm trùng thì sao?#

MongoDB có unique index trên user_id + property_id.

Câu 26: Một booking có thể review nhiều lần không?#

Unique index theo user_id + booking_id ngăn một user review lặp lại cùng booking.

Câu 27: Tại sao dùng Asia/Ho_Chi_Minh?#

Server production có thể chạy UTC. Dùng timezone nghiệp vụ giúp ngày check-in, checkout và hoàn tất booking đúng theo ngày Việt Nam.

Câu 28: SharedPreferences dùng làm gì?#

Lưu onboarding, access token, user info, role, ngôn ngữ và trạng thái giao diện nhỏ.

Câu 29: SharedPreferences có an toàn tuyệt đối không?#

Không. Với production, token nên được chuyển sang secure storage.

Câu 30: Điểm khó nhất của dự án là gì?#

Phần khó nhất là giữ dữ liệu booking và payment nhất quán: kiểm tra phòng trống, chống request đồng thời, tính giá ở server và chỉ xác nhận booking khi webhook hợp lệ.

Câu 31: Nếu hai người cùng đặt một phòng?#

Backend dùng lock và kiểm tra số booking overlap trước khi tạo. Request không còn đủ phòng sẽ bị từ chối.

Câu 32: Nếu webhook đến hai lần?#

Code kiểm tra payment đã PAID và trả lại thay vì cập nhật lặp lại, giúp xử lý idempotent một phần.

Câu 33: Nếu payment PAID nhưng booking update lỗi thì sao?#

Đây là rủi ro consistency. Hướng cải thiện là transaction, retry hoặc reconciliation job.

Câu 34: Vì sao cần admin audit?#

Để biết admin nào đã thực hiện hành động gì, trên dữ liệu nào, kết quả ra sao và vào thời điểm nào.

Câu 35: Vì sao không gọi MongoDB từ Flutter?#

Làm vậy sẽ lộ database credential, khó kiểm soát nghiệp vụ và phân quyền. Flutter chỉ gọi backend API.

Câu 36: Search không dấu hoạt động thế nào?#

Property có search_index đã chuẩn hóa. Backend dùng helper tìm kiếm để đối chiếu từ khóa linh hoạt hơn.

Câu 37: App xử lý lỗi mạng thế nào?#

ApiService có timeout 20 giây, bắt ClientException/TimeoutException và chuyển thành thông báo thân thiện. UI có offline, empty và error state.

Câu 38: Tại sao dùng model Dart?#

Model Dart chuyển JSON thành object có kiểu dữ liệu rõ ràng, giúp UI không phải xử lý Map động ở mọi nơi.

Câu 39: Tại sao có is_active?#

Cho phép archive hoặc khóa dữ liệu thay vì xóa vật lý ngay lập tức, bảo vệ quan hệ lịch sử.

Câu 40: Check-in code dùng làm gì?#

Booking có mã check-in riêng để admin tìm hoặc quét khi khách đến nhận phòng.

Câu 41: Khi nào booking completed?#

Sau checkout và khi attendance cho biết khách đã checked-in.

Câu 42: No-show được xử lý thế nào?#

Booking có thể bị cancelled với cancellation_reason = no_show và không hoàn tiền theo logic hiện tại.

Câu 43: Điểm mạnh lớn nhất của StayZ?Điểm mạnh#

Dự án có luồng booking tương đối đầy đủ, backend thật, kiểm tra nghiệp vụ server-side, payment webhook, AI dựa trên database và phân quyền admin.

Câu 44: Hạn chế lớn nhất?Hạn chế#

Khả năng kiểm thử integration chưa đủ sâu, refresh token Flutter chưa hoàn thiện và payment consistency cần cơ chế phục hồi mạnh hơn.

Câu 45: Nếu làm lại sẽ cải thiện gì trước?#

Tôi sẽ ưu tiên secure token storage, refresh-token rotation, transaction/reconciliation cho payment và thêm end-to-end test cho booking.
